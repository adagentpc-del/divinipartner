Build a production ready web application called A3 Partner Portal.

The application is a simple partner intake and internal request management portal for A3 Visual. It must be clean, premium, modern, easy to use, and optimized for fast intake and internal quote preparation. Keep the interface simple and frictionless. Do not over engineer the first version.

Core goal:
Allow an internal admin to create partner specific portal pages for companies such as Move Miami, Swarm, One One Marketplace, Hilton, and future partners. Each partner portal page should be public and shareable. Their vendors or clients can submit event and project needs through that page. The system stores the request, uploads, AI summary, internal summary, recommended upsells, and a downloadable PDF summary. Only the internal admin receives notification emails.

Technical requirements:
Use React, TypeScript, Tailwind, and a clean component system. Use Supabase for database, storage, and admin auth. Public partner pages do not require login. Internal admin pages require auth. Use Resend for outbound email notifications. Notifications must go to the admin's A3 Outlook inbox. Add an internal AI summary generator and a PDF summary generator.

Visual direction:
The design should feel premium, modern, polished, and easy to use. Partner brand should feel primary, with a small A3 partner badge presence. The portal should feel elegant and conversion focused. Use a clean layout, subtle motion, large buttons, strong whitespace, premium cards, and intuitive progress flow. Keep it impressive but not complicated.

Email behavior:
1. On every partner request submission, send a notification email to ADMIN_EMAIL
2. The admin email will be an Outlook mailbox, but the app should not require Microsoft Graph for version 1
3. Use Resend to send the email
4. Set the Reply To header to the submitter's email when available so the admin can reply directly from Outlook
5. Include the partner name, contact name, company name, event name, event date, requested categories, and direct internal admin link to the request detail page
6. Leave submitter confirmation email off by default in version 1, but structure the code so it can be enabled later

Important email architecture requirement:
Design the email service layer so that it can later be swapped from Resend to Microsoft Graph without rewriting the whole app.
