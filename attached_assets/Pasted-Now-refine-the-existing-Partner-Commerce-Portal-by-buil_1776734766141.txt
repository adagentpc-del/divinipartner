Now refine the existing Partner Commerce Portal by building a selective billing execution and invoice workflow layer. The platform already has partners, portal types, products, packages, branding zones, inventory, reservations, fulfillment modes, supplier routing, quote/spec references, quote ingestion/catalog enrichment, financial scaffolding, exports, reconciliation, and communications. This pass should activate billing execution pathways without forcing one universal money model.

Important:
Do not rebuild from scratch.
Work from the existing codebase.
Preserve current architecture and workflows where sensible.
Do not implement a mandatory direct payment gateway for every partner.
Implement selective billing execution so the platform can support different collection models per partner, per event, and per order.

Primary objective for this pass:
Make the system operationally capable of handling multiple billing paths:
- A3 collected
- Alyssa entity collected
- manual invoice
- split payout placeholder
- external payment pending

This pass should support invoice generation, billing records, payment tracking, and partner-specific execution rules, while still allowing “play nice first” rollout with A3 collected as default.

This pass has 9 major priorities:

1. Billing execution architecture
2. Partner billing activation controls
3. Invoice record model and workflow
4. Payment request and payment tracking workflow
5. Billing path by partner, event, and order
6. Internal billing command center
7. Better order-to-billing visibility
8. Role-safe billing UX
9. Documentation and seed data

8. ROLE SAFE BILLING UX
-----------------------------------

Ensure billing visibility is tightly controlled.

Rules:
- Super Admin and Internal Admin can view all billing controls
- Partner Managers may see limited commercial billing info for their partner if appropriate, but not internal margin or commission verification
- Vendor users should not see billing execution controls or unrelated billing data
- Client users should only see client-facing invoice/payment information intended for them

Do not expose internal commission logic or reconciliation notes in client-facing billing screens.
