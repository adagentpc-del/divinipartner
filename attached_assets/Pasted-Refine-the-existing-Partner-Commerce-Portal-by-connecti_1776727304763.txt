Refine the existing Partner Commerce Portal by connecting order fulfillment modes to the inventory reservation system, adding partner facing committed inventory visibility, and building the first true catalog intelligence and quote mapping layer.

Important:
Do not rebuild from scratch.
Work from the existing codebase.
Preserve the current inventory and reservation engine.
Use what is already live and extend it.
Implement real workflow connections, not placeholders only.

Primary objective for this pass:
Make the ordering system operationally intelligent.
When a client or partner chooses a fulfillment mode, the system should understand whether to reserve existing partner inventory, create print demand only, or flag new hardware requirements.
Also give partner managers a clear view into what inventory is already committed so they can plan confidently.

This pass has 3 major priorities:

1. Wire order fulfillment mode to inventory and reservation behavior
2. Add partner facing committed inventory visibility
3. Build the first catalog intelligence and quote mapping layer

-----------------------------------
1. ORDER FULFILLMENT MODE LOGIC
-----------------------------------

Extend the ordering flow and order data model so each ordered item can have a fulfillment mode that drives operational logic.

Supported fulfillment modes:
- full unit
- graphic only
- use existing partner inventory
- rental hardware plus new print
- new hardware required

Requirements:
- these modes should be selectable in the ordering flow using clickable cards or segmented controls, not clunky dropdowns
- only show applicable fulfillment modes for a product based on product capabilities
- example: if a product is not compatible with graphic only, do not show that option
- example: if a product is not rental eligible, do not show rental hardware plus new print

For each ordered line item, store:
- fulfillment mode
- quantity
- linked product
- hardware required yes/no
- print demand quantity
- hardware demand quantity
- reserved inventory quantity if applicable
- inventory source city/location if applicable
- shortage quantity if applicable
- inventory reservation status
- internal fulfillment notes

Operational behavior:
- graphic only should create print demand only and should not reserve hardware
- use existing partner inventory should check inventory in the chosen city/location and create a reservation if available
- rental hardware plus new print should create hardware demand and print demand, and optionally tie to rental inventory if later supported
- new hardware required should create hardware demand and no reservation from partner inventory
- full unit should behave according to product rules and whether inventory exists or must be newly sourced

If available inventory is insufficient:
- show shortage clearly before final order submission
- allow internal admin or partner manager to proceed with a shortage flag if needed
- store the shortage on the order detail page
- surface shortage in internal ops views

Add inheritance and defaults:
- default inventory source city from event city
- allow override if needed
- preserve user choices when navigating between ordering steps

Add an event commitments view:

Add a committed inventory timeline or upcoming commitments list if feasible:
