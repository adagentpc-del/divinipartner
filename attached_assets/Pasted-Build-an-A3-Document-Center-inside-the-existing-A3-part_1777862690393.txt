Build an A3 Document Center inside the existing A3 partnership portal.

Goal:
Create a secure document management and customer document delivery system where A3 admins can upload documents such as W9, ACT docs, COI, insurance certificates, capability sheets, vendor packets, and customer support documents. Admins should be able to press a button from a lead, quote, partner, or customer record to send selected documents to a customer. Customers should also be able to request or download approved documents, and every send, view, request, and download must be tracked.

Important:
Do not store uploaded documents in the codebase or public static folder.
Use object storage.
Use Replit Object Storage if it is already configured.
If Supabase Storage is already configured and easier based on the current project structure, use Supabase Storage.
Use signed URLs for private documents.
Do not create permanent public links for W9, COI, ACT, insurance, compliance, or vendor setup documents.

Branding:
Keep the design consistent with the existing A3 partnership portal.
Use the existing admin dashboard styling, cards, tables, buttons, modals, badges, and form patterns wherever possible.
Do not redesign the portal.
Add this as a clean operational module.

Feature name:
A3 Document Center

Core document types:
1. W9
2. ACT docs
3. Articles, registration, or corporate documents
4. Certificate of Insurance
5. Insurance certificate
6. Capability sheet
7. Vendor onboarding packet
8. Product guide
9. Artwork upload guide
10. Installation guide
11. Partner packet
12. Customer support docs
13. Internal only documents

Document permission levels:
Create three document visibility levels.

1. public_sales
Used for capability sheets, product guides, general sales documents.
These can be customer downloadable if enabled.

2. customer_requestable
Used for W9, COI, ACT docs, insurance certificates, company registration docs, and vendor onboarding packets.
Customers can request these, but they should not automatically see all of them unless assigned, approved, or configured as auto_send.

3. internal_only
Used for pricing sheets, supplier cost sheets, commission docs, internal routing docs, vendor notes, and documents that should never be customer-facing.

Security requirements:
- Use signed URLs for private docs
- Never expose storage keys directly to the frontend
- Never make W9, COI, ACT, insurance, registration, or vendor docs publicly accessible
- Validate file types server-side
- Validate file size server-side
- Restrict admin document management routes to authenticated admins only
- Restrict customer document access to assigned documents or valid signed links
- Prevent internal_only docs from being customer assigned or downloaded
- Add server-side checks before every download
- Log download attempts where possible
- Do not leak private document URLs in logs

2. Reuse existing patterns wherever possible.
3. Do not duplicate authentication logic.
4. Do not create a second admin dashboard.
5. Do not break existing lead, quote, partner, customer, or upload flows.
6. Add migrations safely.
