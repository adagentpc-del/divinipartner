Now refine the existing Partner Commerce Portal by building a robust supplier routing and fulfillment workflow. The platform already has partners, portal types, products, packages, zones, quotes/spec references, financial scaffolding, inventory, reservations, and ordering flows. This pass should make supplier assignment, fulfillment tracking, and ops handoff materially stronger.

Important:
Do not rebuild from scratch.
Work from the existing codebase.
Preserve the current schema and workflows where sensible.
Implement real workflow improvements, not placeholders only.

Primary objective for this pass:
Make the system capable of routing orders and line items to the correct supplier, tracking supplier execution, and giving internal admins a clear fulfillment command center.

This pass has 7 major priorities:

1. Supplier routing model
2. Supplier assignment by order and by line item
3. Supplier fulfillment views and status tracking
4. Internal fulfillment command center
5. Supplier packet quality
6. Routing intelligence and defaults
7. Better ops notes, exceptions, and handoff clarity

-----------------------------------
1. SUPPLIER ROUTING MODEL
-----------------------------------

Extend the data model to support supplier assignment at multiple levels.

Requirements:
- suppliers already exist as entities; deepen their operational use
- allow supplier assignment at:
  - product level default supplier
  - package component level
  - branding zone level default supplier
  - order level default supplier
  - order line item level override supplier

This matters because a single order may involve:
- one supplier for printing
- another for hardware
- another for fabrication
- another for installation

At minimum, build clear support for:
- order level supplier
- line item level supplier
- inherited defaults from product/package/zone where available

For each line item, store:
- assigned supplier
- supplier assignment source
  - inherited from product
  - inherited from package
  - inherited from zone
  - inherited from order
  - manually overridden
- supplier status
- supplier due date
- supplier notes
- supplier reference number placeholder
- fulfillment exception flag

Show the assignment source clearly in internal admin views.

2. SUPPLIER ASSIGNMENT UX
-----------------------------------

Build a clean supplier assignment experience for internal admins.

Requirements:
- on order detail page, allow assigning supplier at the order level
- also allow assigning supplier per line item
- allow bulk assign supplier for multiple selected line items
- show current supplier on each item
- show inherited default supplier when not yet manually assigned
- allow internal admin to override supplier with confirmation modal
- show clear badges for:
  - unassigned
  - inherited
  - manually assigned
  - exception

Use a polished drawer, modal, or inline editor.
Do not make this a clunky raw form.

For line items, also show useful supplier context if available:
- supplier capabilities
- territory coverage
- linked quote/spec references
- lead time
- default notes

- in production
- awaiting assets

- delivered
- installed
- completed
- issue flagged

Requirements:

- add timestamp fields for key milestones if feasible
- show status progression visually using badges, timeline, or step indicators

For each line item and/or supplier assignment record, allow:
- due date
- expected ship date
- expected delivery date
- install date if relevant
- notes
- issue flag
- issue reason

4. SUPPLIER FULFILLMENT VIEW
-----------------------------------

Deepen the vendor-facing experience.

Each vendor user should be able to:
- view assigned orders
- view assigned line items

- view files/assets that are visible to them

- update allowed statuses
- leave fulfillment notes

Vendor users must not see:

- expected commission

Create a clean vendor dashboard with:
- due soon

- active production
- recent updates
- issues flagged

7. ROUTING INTELLIGENCE AND DEFAULTS
-----------------------------------

Make the system smarter about default routing.

Use available existing data such as:
- product default supplier
- branding zone default supplier
- supplier capabilities
- supplier territory coverage
- lead time
- quote/spec references

Build a first-pass recommendation system:
- when assigning supplier, suggest likely suppliers based on defaults and context
- show “recommended supplier” badges where applicable
- for line items with a product default supplier, prefill that supplier unless overridden
- for zones, inherit zone supplier if configured
- for packages, inherit component suppliers or fall back to order supplier

Do not over-engineer this.
A simple, understandable recommendation layer is enough.
